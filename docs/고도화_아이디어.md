## 0. 현재 프로그램이 하는 일 (한 줄 요약)

`.xlsx`를 `.zip`으로 이름만 바꿔 압축을 푼 뒤, 안쪽 XML에서 정규식으로
`<sheetProtection ...>` 태그를 지우고 다시 압축해 `.xlsx`로 되돌린다.
→ **오직 "시트 보호(sheet protection)"만** 해제한다.

### 코드 구조
| 파일 | 역할 |
|---|---|
| `Form1.cs` | 전 로직 (파일 선택 → 압축 해제 → 태그 제거 → 재압축) |
| `RemovePasswordFromExcel()` | 메인 파이프라인 |
| `ProcessXmlFiles()` | 정규식으로 `<sheetProtection>` 삭제 |
| `CheckProtectionSheet()` | 보호시트 존재 여부 확인 (최근 커밋에서 추가) |

## 1. 🔴 먼저 고쳐야 할 버그 / 위험 (기능 확장보다 우선)

### 1-1. **[논리 버그] 보호 여부 판정이 항상 "없음"으로 나온다**
`RemovePasswordFromExcel()` 안에서 호출 순서가 뒤집혀 있다.

```csharp
ProcessXmlFiles(extractPath);          // ← 여기서 <sheetProtection>를 이미 다 지움
if (!CheckProtectionSheet(extractPath))// ← 그 다음에 "남아있나?" 검사 → 항상 false
{
    label3.Text = "보호시트가 없습니다";     // 항상 이 분기로 감
}
else
{
    label3.Text = "보호시트 비밀번호 해제 완료"; // 사실상 도달 불가(dead code)
    // 파일 열기 안내(Process.Start)도 여기 있어서 절대 실행 안 됨
}
```

**결과**: 실제로 보호가 걸려 있던 파일도 해제는 되지만, 화면에는
"보호시트가 없습니다"라고 잘못 뜨고, "작업 완료 — 파일 열까요?" 안내가 나오지 않는다.

**수정**: 순서를 뒤집는다. `CheckProtectionSheet()`(존재 확인)를 **먼저** 실행해
결과를 저장 → 없으면 조기 종료 → 있으면 `ProcessXmlFiles()`로 제거.

```csharp
bool hadProtection = CheckProtectionSheet(extractPath);
if (!hadProtection) { /* "보호시트가 없습니다" 후 정리/복원 */ return; }
ProcessXmlFiles(extractPath);          // 제거
// "해제 완료" + 열기 안내
```

### 1-2. **[데이터 파괴 위험] 원본을 in-place로 rename 한다**
```csharp
File.Move(excelFilePath, zipFilePath); // 원본 .xlsx 를 .zip 으로 이동(=원본 사라짐)
```
중간에 압축 해제/재압축이 실패하면 **원본이 `.zip` 상태로 남아 복구가 어렵다.**
→ **원본은 절대 건드리지 말고 복사본에서 작업**하고, 성공 시에만 결과물을 쓴다.
(`파일명_unlocked.xlsx`로 새로 저장하는 방식 권장 → 원본 보존)

### 1-3. **[finally 예외] 임시폴더 삭제가 오히려 예외를 던질 수 있다**
```csharp
finally { Directory.Delete(extractPath, true); }
```
압축 해제 전에 예외가 나면 `extractPath`가 없어 `finally`에서 또 예외.
Excel/백신이 파일을 잡고 있으면 삭제 실패.
→ `if (Directory.Exists(extractPath))` 가드 + try/catch, 그리고 임시폴더를
**원본 옆("임시폴더")이 아니라 `%TEMP%` 아래 GUID 폴더**에 만든다(사용자 폴더 오염 방지).

### 1-4. **[유지보수/보안] 안 쓰는 무거운 의존성 4종**
`EPPlus 4.1`, `Microsoft.Office.Interop.Excel`, `OfficeOpenXml.Core`,
`OfficeOpenXml.Extends` — 참조만 걸려 있고 실제 코드는 `Ionic.Zip`만 쓴다.
특히 **EPPlus 4.x는 알려진 취약점이 있는 구버전**이라 배포물에 넣을 이유가 없다.
→ 미사용 참조/패키지 전부 제거.

### 1-5. **[정확성] 정규식으로 XML을 편집한다**
`Regex.Replace(content, "<sheetProtection[^>]*>", "")` 는
자체 종료 태그(`<sheetProtection ... />`)나 예외적 포맷에서 어긋날 수 있고,
XML 편집을 문자열 치환으로 하는 건 원칙적으로 취약하다.
→ `System.IO.Packaging`(OPC) 또는 `System.Xml.Linq`로 **워크시트 파트만**
정확히 파싱해 `sheetProtection` / `workbookProtection` 요소를 제거.

## 2. 🟢 "최신 엑셀 지원" — 실제로 무엇을 더 지원해야 하나

> 최신 Microsoft 365 / Excel 2024도 `.xlsx`는 여전히 OOXML(zip) 구조라
> **시트 보호 해제는 지금 방식으로 최신 버전까지 이미 동작한다.**
> 진짜 "미지원" 영역은 아래다.

### 2-1. 보호의 3가지 층위 — 지금은 1층만 처리
| 보호 종류 | 저장 위치 | 암호화? | 현재 지원 | 난이도 |
|---|---|---|---|---|
| **시트 보호** (셀 편집 잠금) | `xl/worksheets/sheetN.xml` `<sheetProtection>` | ❌ 해시만 | ✅ 지원 | 쉬움 |
| **통합문서 보호** (시트 추가/삭제/이동 잠금) | `xl/workbook.xml` `<workbookProtection>` | ❌ 해시만 | ❌ 미지원 | 쉬움 |
| **파일 열기 암호** (문서 전체 암호화) | CFB/OLE 컨테이너 (`EncryptionInfo` + `EncryptedPackage`) | ✅ AES-256/SHA-512 | ❌ 미지원 | 어려움 |

- **(a) 통합문서 보호 추가** — `<workbookProtection>` 태그도 함께 제거하면
  "시트 구조 잠금"까지 풀린다. **거의 공짜 확장(정규식/파서 한 줄 추가).**
- **(b) 파일 열기 암호** — 이건 근본적으로 다르다. 열기 암호가 걸린 파일은
  **zip이 아니라 CFB(OLE2) 컨테이너**라서 현재 `ZipFile.Read`가 바로 예외를 던진다.
  - 정당한 지원 방법: **사용자에게 암호를 입력받아 복호화** → 내부 zip 추출.
    (`NPOI`의 `EncryptionInfo`/`Decryptor`, 또는 `msoffcrypto` 계열 로직)
  - 암호를 모르는 경우의 무차별 대입(brute-force)은 **본인 소유/권한 있는 파일에
    한정**해야 하며 별도 정책·경고가 필요. 기본 기능으로 넣지 말 것.

### 2-2. 파일 포맷 확장
| 포맷 | 설명 | 현재 | 확장 방법 |
|---|---|---|---|
| `.xlsm` | 매크로 포함 통합문서 | ❌ (파일 필터에서 막힘) | 구조는 xlsx와 동일 → **파일 필터만 넓히면 시트 보호 해제 즉시 동작** |
| `.xltx`/`.xltm` | 서식 파일 | ❌ | 위와 동일(OOXML zip) |
| `.xls` | 구형 BIFF8 이진 (Excel 97–2003) | ❌ | `NPOI`로 처리 가능(별도 코드 경로 필요) |
| `.ods` | OpenDocument | ❌ | zip 구조지만 태그 다름, 수요 낮음 |

- **가장 가성비 높은 확장**: 파일 열기 대화상자 필터를
  `*.xlsx;*.xlsm;*.xltx;*.xltm`로 넓히는 것. 코드 로직 변경 거의 없이 지원 포맷이 늘어난다.

### 2-3. VBA 프로젝트 암호 (`.xlsm`)
매크로 편집기 잠금은 `xl/vbaProject.bin` 안의 `DPB`/`CMG`/`GC` 키에 저장.
바이트를 무력화하면 편집기 잠금 해제 가능(고급 기능, 선택).

## 3. 🟡 UX / 기능 고도화

1. **드래그 앤 드롭** — 파일을 창에 끌어다 놓으면 바로 선택. (`AllowDrop`, `DragDrop` 이벤트)
2. **일괄 처리(batch)** — 여러 파일 / 폴더 통째로 선택해 한 번에 해제 + 진행률 표시.
3. **비동기 처리** — 지금은 UI 스레드에서 동기 실행이라 큰 파일에서 창이 멈춘다.
   `async/await` + `IProgress<T>`로 진행바 표시.
4. **원본 보존 옵션** — 기본은 `원본명_unlocked.xlsx` 새 파일 생성, 체크박스로
   "원본 덮어쓰기" 선택. (1-2 위험 해소와 연결)
5. **결과 로그 패널** — 처리한 파일 / 제거한 보호 종류(시트·통합문서·개수)를 목록으로.
6. **암호 입력 다이얼로그** — 열기 암호 지원 시 필요.
7. **다국어(i18n)** — 현재 한글 하드코딩. 리소스 분리로 영어 등 추가.
8. **UI 현대화** — 버튼/라벨 이름(`button1`, `label3`)이 자동생성명 그대로.
   의미 있는 이름 + 레이아웃 정리 + 하이 DPI 대응.

## 4. 🔧 기술 부채 / 현대화

1. **Ionic.Zip(DotNetZip) 제거** — 오래 방치된 라이브러리.
   .NET 내장 `System.IO.Compression.ZipFile` / `System.IO.Packaging`로 대체 →
   외부 의존성 0, OPC 구조 보존이 더 안전.
2. **미사용 의존성 정리** (1-4) — EPPlus 4.1 / Interop.Excel / OfficeOpenXml 제거.
3. **.NET 8 로 마이그레이션(선택)** — WinForms는 .NET 8에서 계속 지원.
   단일 실행 파일(self-contained single-file) 배포 → .NET 설치 없이 실행 가능.
   (또는 프레임워크 의존 배포로 용량 절감)
4. **관심사 분리** — 해제 엔진을 `Form1.cs`에서 분리해
   `UnlockService`/`ExcelUnlocker` 클래스로. 콘솔/CLI 재사용 및 단위 테스트 가능.
5. **단위 테스트** — 보호 걸린 샘플 `.xlsx`/`.xlsm`를 리소스로 넣고
   "해제 후 `<sheetProtection>` 없음 + 파일이 정상 열림"을 검증.
6. **로깅** — 실패 원인(잠긴 파일, 암호화 파일 등) 진단용 로그.

## 부록: "최신 엑셀"에 대한 오해 정리
- 최신 Excel도 `.xlsx`는 **여전히 OOXML(zip)** → 시트 보호 해제는 지금 방식으로 **이미 최신 대응됨**.
- 진짜 못 여는 건 **"파일 열기 암호(전체 암호화)"** 뿐 → 이건 zip이 아니라
  **CFB 컨테이너 + AES 암호화**라 별도 복호화 경로가 필요하다.
- 따라서 "최신 엑셀 지원"의 실질 = ① 통합문서 보호 + ② 매크로/서식 확장자 +
  ③ (선택) 열기 암호 복호화.
